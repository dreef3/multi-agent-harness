# Comprehensive Audit: Multi-Agent Harness

**Date**: 2026-03-28
**Scope**: Architecture, security, dead code, test strategy, feasibility, documentation

## Context

This is a full audit of the multi-agent-harness project â€” a system that orchestrates AI coding agents via Docker containers, with a planning agent (master) that decomposes work into tasks executed by sub-agents. The system uses Express/SQLite on the backend, React/Vite on the frontend, and communicates via WebSocket, TCP RPC, and REST.

---

## 1. Architecture Assessment

### Strengths

- **Clean separation of concerns**: orchestrator (`taskDispatcher`, `recoveryService`, `containerManager`, `planningAgentManager`), store layer (SQLite), API layer (Express routes), connectors (GitHub/Bitbucket abstraction), and agents (planning/sub-agent runners).
- **Well-designed state machine**: Project lifecycle (`brainstorming -> spec -> plan -> executing -> completed/failed`) is clearly modeled in `backend/src/models/types.ts`.
- **Robust recovery**: `RecoveryService` handles stale sessions, orphaned projects, and retry logic with concurrency semaphores (global + per-project slots). Boot recovery scans for incomplete work.
- **Good container isolation**: Sub-agents run in Docker with memory/CPU limits, credential cleanup (GIT_PUSH_URL deleted before AI starts), and dedicated network.
- **VCS abstraction**: `VcsConnector` interface (`backend/src/connectors/types.ts`) cleanly abstracts GitHub and Bitbucket Server, making it easy to add providers.
- **Observability hooks**: OpenTelemetry instrumentation for task dispatch, active agents, and spans. Sub-agent event forwarding for real-time UI updates.

### Concerns

1. **Singleton pattern overuse** â€” `DebounceEngine`, `RecoveryService`, and `PlanningAgentManager` all use module-level `let instance` + getter/setter. This makes testing harder (must mock globals), prevents running multiple instances, and creates hidden coupling. Consider dependency injection via a service container or constructor parameters in `index.ts`.

2. **TCP RPC is fragile** â€” The planning agent uses a custom newline-delimited JSON-RPC over raw TCP (`planning-agent/runner.mjs:320-360`). There's no framing protocol, no message IDs for request-response correlation, and no built-in keepalive. The `process.stdout.write` override (`runner.mjs:337-345`) is clever but brittle â€” any library writing to stdout will corrupt the RPC stream.

3. **Polling-based container completion** â€” `TaskDispatcher.waitForCompletion()` (`taskDispatcher.ts:254-303`) polls every 5 seconds via `setInterval`. Docker events API (`watchContainerExit` at `containerManager.ts:134`) exists but isn't used for task completion. This wastes cycles and adds 0-5s latency to task completion detection.

4. **Dynamic imports in hot paths** â€” `taskDispatcher.ts:310` does `await import("../store/agents.js")` inside `getSessionStatus()` which is called every 5 seconds per active task. This is a static module that should be imported at the top level.

5. **`harness-agents` network is not internal** â€” `docker-compose.yml:60` sets `internal: false` on the agents network, meaning sub-agent containers can reach the internet. This is necessary for git push/clone but increases attack surface if a sub-agent is compromised.

6. **No graceful shutdown** â€” `backend/src/index.ts` has no SIGTERM/SIGINT handler. Active WebSocket connections, running containers, and in-flight TCP RPC calls will be dropped without cleanup.

### Critical

*None found* â€” `.env` files with secrets are properly gitignored (only `.env.example` and `e2e-tests/.env.e2e` are tracked, both contain no secrets).

| S7 | **Webhook signature verification uses `JSON.stringify(req.body)`** | `webhooks.ts:63` | If Express JSON parser normalizes whitespace/ordering differently from the raw payload, signature check may fail or be bypassable. Should verify against raw body buffer. |

## 6. Feasibility Assessment

### What Works Well

The system is **functional and production-tested** for its intended use case: orchestrating AI coding agents on a single host. The core loop (plan -> dispatch -> execute -> PR) is well-implemented. Recovery from crashes, stale sessions, and partial failures is robust. Multi-provider support (6 agent providers, 2 VCS providers) shows good extensibility.

### Scaling Limits

| Dimension | Current Limit | Bottleneck |
|-----------|--------------|------------|
| Concurrent sub-agents | 3 (configurable) | Docker host memory/CPU |
| Projects | Unbounded | SQLite write contention under high concurrency |
| Planning agents | 1 per project | TCP port allocation, container overhead |
| WebSocket connections | Unbounded per project | Node.js event loop, memory |

### Risks

1. **SQLite under concurrency**: WAL mode helps, but multiple projects dispatching tasks simultaneously will contend on writes. Not a problem at current scale (<10 concurrent projects) but would need migration to Postgres/etc. for growth.
2. **Single-host Docker**: All containers run on one Docker daemon. No orchestration for multi-host scaling (no Kubernetes/Swarm).
3. **pi-coding-agent SDK dependency**: The entire agent runtime depends on `@mariozechner/pi-coding-agent`. This is an external dependency from a GitHub fork (`superpowers` package from `github:obra/superpowers#v5.0.4`). Breaking changes or abandonment would be impactful.

## 7. Prioritized Recommendations

### Immediate (High Impact, Low Effort)

1. **Remove dead code**: Delete `TaskDispatcher.activeTasks`/`getActiveTaskCount()`, simplify `DebounceStrategy` type, remove `preInitAgent` no-op.
2. **Fix dynamic import**: Move `import("../store/agents.js")` in `taskDispatcher.ts:310` to top-level static import.
3. **Add frontend tests to CI**: Add `vitest run` step to `ci.yml` for the frontend workspace.
4. **Write a README**: Project description, setup instructions, architecture overview.

### Short-term (High Impact, Medium Effort)

5. **Add basic auth middleware**: Even a shared secret/API key would dramatically improve security posture for non-localhost deployments.
6. **Add input validation**: Use the already-installed `@sinclair/typebox` (it's in `package.json`) to validate API request bodies via middleware.
7. **Add security headers**: Install `helmet` for Express, add headers in `nginx.conf`.
8. **Use Docker events instead of polling**: Replace `waitForCompletion()` interval with `watchContainerExit()` which already exists in `containerManager.ts:134`.
9. **Add webhook signature test**: Verify the `verifySignature` function and raw body handling.
10. **Test untested frontend pages**: `PlanApproval`, `PrOverview`, `Settings`.

### Medium-term (Strategic)

11. **Add graceful shutdown**: SIGTERM handler in `index.ts` that drains WebSocket connections, stops polling, and allows in-flight tasks to complete.
12. **Harden container security**: Add `cap_drop: ALL`, `security_opt: [no-new-privileges:true]`, and `read_only: true` (with tmpfs for writable paths) to sub-agent containers.
13. **Restrict Docker socket proxy**: Remove IMAGES and NETWORKS permissions; the backend only needs CONTAINERS + POST.
14. **Replace singleton pattern**: Inject services via constructor/factory in `main()`, improving testability.
15. **Add structured logging**: JSON-formatted logs with correlation IDs (projectId, sessionId, taskId).

### Long-term (If Scaling)

16. **Migrate SQLite to PostgreSQL**: For concurrent multi-project workloads.
17. **Add container orchestration**: Kubernetes or Docker Swarm for multi-host scaling.
18. **Replace TCP RPC with gRPC or HTTP**: More robust framing, built-in keepalive, request-response correlation.

## 8. pi-coding-agent SDK Integration

The entire agent runtime is built on [@mariozechner/pi-coding-agent](https://github.com/badlogic/pi-mono/tree/main/packages/coding-agent) `^0.61.1`. All three packages (backend, planning-agent, sub-agent) depend on it.

### What's Working Well

- **Correct RPC mode usage**: Planning agent implements TCP bridge over pi's JSONL RPC protocol
- **Guard hooks**: Both agents use `bash.spawnHook` to block destructive operations
- **Credential cleanup**: Tokens deleted from `process.env` before AI session starts
- **Event forwarding**: Sub-agent subscribes to session events for real-time UI updates
- **Custom tools**: Well-structured domain tools using TypeBox schemas
- **Extension system**: Output filter extension properly reduces token usage

### Concerns

| # | Finding | Severity |
|---|---------|----------|
| SDK1 | **Caret version range** (`^0.61.1`) â€” pre-1.0 semver allows breaking changes in minor bumps | Medium |
| SDK2 | **`web_fetch` tool duplicated** in both `planning-agent/tools.mjs` and `sub-agent/tools.mjs` â€” bug fixes must be applied twice | Low |
| SDK3 | **Private IP blocking incomplete** â€” blocks RFC1918 IPv4 but not IPv6 (`::1`, `fe80::`) | Low |
| SDK4 | **No error handling around `runRpcMode()`** â€” planning agent runner doesn't catch RPC errors | Medium |
| SDK5 | **Heartbeat sent even when event forwarding fails** â€” backend may think agent is healthy when it's not communicating | Low |
| SDK6 | **`SettingsManager.inMemory()` undocumented choice** â€” all agents ignore persisted pi settings; correct for containers but not documented | Low |
